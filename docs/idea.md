# IntentOS 核心设计文档

## 核心概念

**IntentOS** 是一个基于「Skill 生成 SkillApp」的操作系统。用户无需编写代码，通过简要的意图描述就能将已有的 Skill 实例化为一个完整的、具备交互界面的应用（SkillApp）。系统自动处理设计规划、代码生成、编译打包等全流程，让用户能快速将 Skill 转化为可用的应用。

---

## 架构层级

IntentOS 采用分层架构设计，从上到下共四个层级：应用层、IntentOS 管理层、AI Provider 层（可插拔 AI 后端）和宿主 OS 资源层。应用层内的 IntentOS Desktop 与 SkillApp 平行运行；IntentOS 管理层和 AI Provider 层共同构成 OS 核心，统称 IntentOS 层。各层职责明确，通过清晰的接口进行协作。

```
┌─────────────────────┐  ┌─────────────────────┐
│  IntentOS Desktop   │  │     SkillApp(s)      │  ← 应用层（平行关系）
│      （管理台）      │  │    （各类应用）       │
└──────────┬──────────┘  └──────────┬───────────┘
           └─────────────┬──────────┘
┌──────────────────────────────────────────────┐
│                  IntentOS 层                  │  ← OS 核心层
│  ┌──────────────────┐  ┌───────────────────┐  │
│  │     管理层        │  │   AI Provider 层  │  │
│  │ (桌面管理/生命周期)│  │  (可插拔 AI 后端) │  │
│  └──────────────────┘  └───────────────────┘  │
└──────────────────────────────────────────────┘
               │ 访问
┌──────────────▼───────────────────────────────┐
│               宿主 OS 资源层                   │
│        MCP / 文件系统 / 网络 / 进程            │
└──────────────────────────────────────────────┘
```

### 各层职责说明

**应用层** — 用户直接交互的界面，包含两个平行组件：
- **IntentOS Desktop（管理台）**：OS 风格的桌面管理界面，负责 Skill 管理、SkillApp 管理和生成流程交互
- **SkillApp(s)**：由系统自动生成的各类独立应用，与管理台平行运行

**IntentOS 管理层** — 整个桌面环境和所有 SkillApp 生命周期的 OS 级调度层：
- 协调 IntentOS Desktop 与各 SkillApp 之间的关系
- 管理应用启动、退出、窗口调度等 OS 级行为

**AI Provider 层（可插拔 AI 后端）** — AI 执行内核，是规划与生成的大脑：
- 通过 AIProvider 抽象接口，统一规划引擎与代码生成引擎的调用方式
- 将用户意图转化为可运行的 SkillApp 代码
- MVP 实现为 Claude API Provider，后续可无缝接入 OpenClaw 等本地 Provider

**宿主 OS 资源层** — 底层能力和资源的统一接口：
- 提供 MCP 工具和接口
- 暴露文件系统、网络、进程等宿主操作系统能力
- 支撑应用层和 IntentOS 层的实际运行环境

## 核心组件

### 1. IntentOS Desktop（管理台）

**定义**：基于 Electron 构建的桌面管理平台，是用户与整个系统的主要交互界面。与 SkillApp 处于同一应用层，平行运行。

**边界与职责**：
- **Skill 管理中心** — 展示已安装的 Skill，支持联网浏览和下载新 Skill
- **SkillApp 管理中心** — 列表展示所有生成的应用，支持启动、卸载、修改等操作
- **应用生成窗口** — 承载从 Skill 到 SkillApp 的交互式生成流程

**不涉及**：代码生成、Skill 执行、资源底层操作、SkillApp 生命周期调度（由 IntentOS 管理层负责）

### 2. SkillApp

**定义**：由 AI Provider 层自动生成的 Electron 应用。每个 SkillApp 是独立的、完整的、可直接使用的应用，与 IntentOS Desktop 平行运行于应用层。

**边界与职责**：
- **独立应用** — 基于 Electron 构建，拥有独立的窗口、进程和生命周期
- **能力调用** — 通过 AI Provider 层提供的接口调用一个或多个 Skill，获得计算能力
- **资源访问** — 可通过 MCP 和宿主 OS 接口访问文件、网络、进程等资源
- **动态扩展** — 支持增量修改（追加新 Skill、调整界面等），并通过热更新来更新自身代码

**生命周期**：
1. 首次生成 — 从 Skill 到完整应用
2. 运行阶段 — 用户与 SkillApp 交互，调用 Skill
3. 修改升级 — 在管理中心提出需求，系统重新规划并热更新
4. 卸载 — 从系统中移除应用及其相关资源

### 3. IntentOS 层（OS 核心层）

IntentOS 层是整个系统的核心，向上管理应用层，向下访问宿主 OS 资源。分为两个子层：

**管理层**

**定义**：负责管理整个桌面环境和所有 SkillApp 生命周期的 OS 级调度层。

**边界与职责**：
- 协调 IntentOS Desktop 与各 SkillApp 之间的关系
- 管理应用启动、退出、窗口调度等 OS 级行为
- 作为应用层与内核层之间的中间协调者

**AI Provider 层（可插拔 AI 后端）**

**定义**：运行在宿主 OS 上的 AI 执行核心，是整个系统的规划与生成大脑。通过 AIProvider 抽象接口屏蔽底层 AI 实现，系统不绑定特定 AI 后端。

**边界与职责**：
- **规划引擎** — 根据用户意图和 Skill 能力，生成 SkillApp 的初步设计方案（页面布局、交互逻辑、功能分配）
- **代码生成引擎** — 将设计方案转化为可编译的 Electron 应用代码
- **Skill 执行环境** — 在 SkillApp 运行时，负责调用、管理已安装的 Skill 实例

**Provider 实现**：
- **Claude API Provider（MVP）** — 使用 `@anthropic-ai/sdk` 和 `@anthropic-ai/claude-agent-sdk`，快速验证核心生成流程
- **OpenClaw Provider（后续迭代）** — 接入 OpenClaw 本地模型，实现离线可用和更低延迟

**不涉及**：UI 渲染、用户交互、应用窗口管理

## 关键交互流程

### 生成流程（安装态）

从选择 Skill 到生成可用应用的完整过程：

```
1. 用户在 Skill 管理中心选择一个 Skill
         ↓
2. IntentOS Desktop 弹出「SkillApp 生成窗口」（类安装向导）
         ↓
3. AI Provider 层后台分析 Skill 能力与参数，生成初步 App 设计方案
   （此步骤用户不感知，系统内部完成）
         ↓
4. 生成窗口展示设计方案和配置选项供用户选择/调整
   （可支持多轮交互与完善）
         ↓
5. 用户审核无误后点击「生成」按钮，开始正式构建流程
         ↓
6. 进度条推进，依次进行代码生成 → 编译 → 打包
   （AI Provider 层生成代码 → 调用 Node.js/Electron 工具链编译 → 产生可执行文件）
         ↓
7. 生成完成，窗口「原地变形」成为可用的 SkillApp
   （窗口内容从「生成向导」切换为「应用主界面」，用户可立即开始使用）
```

**关键特性**：
- **交互式规划** — 生成窗口不仅是进度显示，更是用户与 AI 规划的协作界面
- **渐进式生成** — 用户可逐步调整配置，系统支持多轮交互后再生成
- **无缝过渡** — 生成完成后无需新建窗口，直接在原窗口中变形为应用，减少认知和操作负担

### 修改流程（运行态）

已生成的 SkillApp 在使用过程中的升级与扩展：

```
用户在 SkillApp 管理中心提出修改需求
       ↓
AI Provider 层分析现有 SkillApp 代码与配置
       ↓
根据需求生成增量方案（增加新 Skill、调整布局、更新逻辑等）
       ↓
动态重新生成对应模块
       ↓
热更新 SkillApp，用户无需重启即可使用新版本
```

## 关键设计理念

### 1. Intent-First（意图优先）

**核心思想**：用户只需描述 *想做什么*（意图），系统负责 *怎么做*（实现）。

用户无需了解代码细节、架构设计、技术选型，只需用自然语言或简单配置表达需求。AI Provider 层将其转化为完整的应用设计和代码。这让非技术用户也能快速创建应用。

### 2. Skill 即原子单元

**核心思想**：每个 Skill 是一个独立的、可组合的功能原子。

Skill 是系统中最小的可复用单位，定义明确的输入、处理逻辑和输出。一个 SkillApp 可以组合多个 Skill，不同应用可以共享和重用同一个 Skill。这种原子化设计提高了系统的灵活性和扩展性。

### 3. 窗口即过程

**核心思想**：生成窗口不仅是 UI 等待框，更是规划与协作的交互界面。

生成窗口承载完整的 Skill-to-App 转化过程。用户可以在窗口内与 AI 多轮交互、调整设计方案、预览效果，充分参与规划过程。这提升了用户的参与感和满足度，也确保生成的应用符合用户期待。

### 4. 原地变形

**核心思想**：生成完成后，窗口直接变形为应用，而不是打开新窗口。

这减少了认知转换和操作断层。用户的注意力和上下文保持连贯，从「生成界面」自然过渡到「应用主界面」，整个体验更加流畅和沉浸。

### 5. 可插拔 AI 后端

**核心思想**：通过 AIProvider 抽象层，系统不绑定特定 AI 后端。

MVP 阶段使用 Claude API 快速验证核心流程，无需本地部署即可启动；后续可无缝切换到 OpenClaw 等本地模型，实现离线可用、更低延迟和更强的数据隐私保障。这种设计让技术选型与产品核心逻辑解耦，随时可以替换或扩展底层 AI 能力。

### 6. 动态生长

**核心思想**：SkillApp 不是一次性产物，而是可持续扩展和进化的活体。

已生成的应用可以根据用户反馈和实际使用情况持续增强功能、优化性能。用户可以随时在管理中心提出修改需求，系统会增量规划和重新生成，使应用不断迭代和完善。

## 技术选型

| 层级 | 技术栈 | 说明 |
|------|--------|------|
| **应用层 — IntentOS Desktop** | Electron + React/Vue | 跨平台桌面应用框架，支持现代 Web 开发体验 |
| **应用层 — SkillApp** | Electron（动态生成） | 自动生成的应用使用 Electron，确保跨平台一致性 |
| **IntentOS 管理层** | Electron 主进程 | 负责桌面和 SkillApp 生命周期调度的 OS 级管理 |
| **AI Provider 层（MVP）** | Claude API（`@anthropic-ai/sdk`）+ Claude Agent SDK（`@anthropic-ai/claude-agent-sdk`） | MVP 阶段使用 Claude API 快速验证核心生成流程 |
| **AI Provider 层（后续）** | OpenClaw Provider（本地模型） | 后续可选接入 OpenClaw 实现离线可用和更低延迟 |
| **宿主 OS 资源层** | MCP + 宿主 OS 接口 | 通过 MCP 协议和系统接口暴露底层能力 |

---

## 工作流总结

IntentOS 的核心工作流可总结为一个 **需求 → 规划 → 生成 → 使用 → 迭代** 的闭环：

1. **需求** — 用户选择 Skill 或提出应用修改需求
2. **规划** — AI Provider 层分析需求，生成设计方案
3. **生成** — 自动生成代码，编译打包
4. **使用** — 用户启动并使用 SkillApp
5. **迭代** — 收集反馈，提出新需求，循环优化

这个流程通过 Intent-First 和 Skill 原子化，最大化了开发效率；通过「窗口即过程」和「原地变形」，最大化了用户体验；通过「动态生长」，确保应用的长期活力；通过「可插拔 AI 后端」，保持技术架构的灵活性与可演进性。
