# IntentOS 用户与需求分析

---

## 1. 目标用户画像

### 画像一：非技术型个人用户——「想用工具但不会编程」

**代表人物**：自由职业者、内容创作者、中小企业主、学生

**技术背景**：
- 日常使用 Office、Notion、浏览器等成熟软件
- 没有编程经验，无法读写代码
- 能流利使用自然语言描述需求

**使用场景**：
- 想把日常重复性工作（如数据整理、文件重命名、内容生成）自动化
- 希望拥有一个「只为自己定制的小工具」，但不想花钱找开发者
- 发现现成 App 的功能不完全符合自己的使用习惯，想微调

**核心诉求**：
- 用中文描述需求，系统自动生成能用的 App
- 不需要理解技术细节，「生成」即可使用
- 应用可以随时改，改完立刻能用

---

### 画像二：个人开发者——「想快速验证想法的独立开发者」

**代表人物**：独立开发者、全栈工程师、技术型产品经理

**技术背景**：
- 熟悉编程，但精力有限，不想为每个小工具都写完整代码
- 了解 Electron、Node.js 等技术栈
- 习惯快速迭代，重视效率

**使用场景**：
- 需要快速搭建内部工具原型（如数据看板、自动化脚本的 GUI 前端）
- 希望把已有的 Skill/脚本快速包装成可视化应用
- 想用最少时间验证产品 idea，再决定是否深入开发

**核心诉求**：
- 通过 Skill 组合快速生成可运行的 Electron 应用
- 生成的应用代码结构清晰，需要时可以手动接管
- 支持热更新，迭代速度快

---

### 画像三：企业内部工具用户——「需要定制化内部工具的非 IT 业务人员」

**代表人物**：运营、销售、HR、数据分析师等业务部门人员

**技术背景**：
- 有一定数字化工具使用能力（Excel、企业系统等）
- 不具备开发能力，依赖 IT 部门或外包
- 清楚自己的业务流程和痛点

**使用场景**：
- 需要一个符合本部门业务流程的专属工具（如批量数据处理、报告自动生成）
- IT 排期紧张，等不到资源，或外包成本太高
- 现有 SaaS 产品功能冗余，想要「刚好够用」的轻量工具

**核心诉求**：
- 自助生成符合业务逻辑的工具应用，不依赖 IT 资源
- 工具能访问本地文件、企业内部数据
- 工具可以随业务变化持续迭代，无需重新采购

## 2. 用户痛点

### 直接写代码的局限

| 痛点 | 描述 |
|------|------|
| **门槛高** | 非技术用户完全无法参与，即使是开发者也需要花大量时间搭环境、写模板代码 |
| **效率低** | 每次需求变更都需要重新编写、测试、打包，周期长 |
| **交付断层** | 从「想法」到「可用应用」之间没有连续流程，需要经过大量中间步骤 |
| **维护成本高** | 工具多了之后，每个工具都需要独立维护，缺乏统一管理 |

### 传统 OS + 现成软件的局限

| 痛点 | 描述 |
|------|------|
| **定制化不足** | 现成软件功能固定，无法精确匹配个人或团队的特定工作流 |
| **Skill 无法复用** | 不同工具间的能力是孤岛，无法跨应用共享和组合 |
| **无意图理解** | 传统 OS 不理解用户意图，所有操作需要用户手动完成 |
| **应用生成能力缺失** | 无法根据自然语言描述自动生成新的应用 |
| **更新割裂** | 工具更新需要用户主动介入，无法基于使用反馈自动迭代 |

## 3. 核心需求列表

### Must Have（MVP 必须有）

| 编号 | 需求 | 说明 |
|------|------|------|
| M01 | **Skill 管理中心** | 用户可查看已安装的 Skill，支持浏览和下载新 Skill |
| M02 | **SkillApp 管理中心** | 列表展示所有已生成的 SkillApp，支持启动、卸载操作 |
| M03 | **自然语言意图输入** | 用户通过自然语言描述需求，系统理解并启动生成流程 |
| M04 | **应用生成窗口（交互式向导）** | 提供引导式的 Skill-to-App 生成交互界面，支持多轮交互调整 |
| M05 | **OpenClaw 规划引擎集成** | 自动分析 Skill 能力，生成 SkillApp 的设计方案（页面布局、交互逻辑） |
| M06 | **代码生成与编译打包** | 将设计方案自动转化为 Electron 应用代码，并完成编译打包 |
| M07 | **原地变形（窗口无缝过渡）** | 生成完成后，生成窗口直接变形为可用的 SkillApp 主界面 |
| M08 | **SkillApp 独立运行** | 每个 SkillApp 作为独立 Electron 应用运行，拥有独立窗口和进程 |
| M09 | **MCP 资源访问** | SkillApp 可通过 MCP 协议访问宿主 OS 的文件系统、网络等资源 |
| M10 | **SkillApp 生命周期管理** | IntentOS 管理层负责 SkillApp 的启动、运行、退出调度 |
| M11 | **生成进度反馈** | 代码生成、编译、打包各阶段向用户提供可见的进度状态 |
| M12 | **Skill 原子化与复用** | 同一 Skill 可被多个 SkillApp 引用和调用，不重复安装 |
| M13 | **SkillApp 热更新** | 修改 SkillApp 后，无需重启即可生效；热更新是修改流程的核心机制，与生成流程同等重要 |
| M14 | **增量修改流程** | 用户在管理中心对已有 SkillApp 提出修改需求，系统增量重新生成对应模块，而非完整重新生成 |

### Should Have（重要但不阻塞 MVP）

| 编号 | 需求 | 说明 |
|------|------|------|
| S03 | **多 Skill 组合生成** | 一个 SkillApp 可组合多个 Skill，生成具有复合能力的应用 |
| S07 | **生成代码可读性保障** | OpenClaw 生成的 SkillApp 代码应遵循清晰的目录结构和命名规范，开发者可阅读、查看和手动接管生成代码 |
| S04 | **设计方案预览与调整** | 生成前向用户展示 SkillApp 的布局和功能方案，支持用户手动调整后再生成 |
| S05 | **Skill 联网市场** ⚠️ 当前版本不含 | 用户可在线浏览和下载社区共享的 Skill |
| S06 | **窗口调度管理** | IntentOS Desktop 支持管理多个 SkillApp 窗口的排列与切换 |

### Nice to Have（锦上添花）

| 编号 | 需求 | 说明 |
|------|------|------|
| N01 | **应用使用数据统计** | 记录 SkillApp 的使用频率、调用次数等，辅助用户决策 |
| N02 | **SkillApp 导出与分享** | 用户可将生成的 SkillApp 导出为独立安装包分享给他人 |
| N03 | **生成历史记录** | 保留每次生成和修改的历史版本，支持回滚 |
| N04 | **Skill 开发者工具** | 提供 Skill 开发和调试工具，方便开发者创建和发布 Skill |
| N05 | **多语言界面支持** | IntentOS Desktop 和生成窗口支持多语言切换 |

## 4. 非功能需求

### 性能

- **生成响应时间**：OpenClaw 完成初步设计方案的时间应在 30 秒内（P90），代码生成+编译完整流程不超过 3 分钟
- **启动时间**：IntentOS Desktop 冷启动时间不超过 5 秒；SkillApp 启动时间不超过 3 秒
- **热更新延迟**：SkillApp 热更新生效时间不超过 10 秒
- **并发运行**：系统应支持同时运行至少 5 个独立 SkillApp，不出现显著性能下降

### 安全

- **Skill 沙箱隔离**：各 SkillApp 运行在独立进程中，相互之间不能直接访问对方的内存和数据
- **资源访问权限控制**：SkillApp 通过 MCP 接口访问系统资源时，需经过用户授权或权限声明，不得静默访问敏感目录
- **代码生成审查**：OpenClaw 生成的代码在编译前应经过基本安全扫描，防止恶意代码注入
- **Skill 来源可信度**：从 Skill 市场下载的 Skill 应标注来源和签名信息，用户可识别可信来源
- **本地数据保护**：用户的意图描述、应用数据默认存储在本地，不上传至外部服务器（明确告知时除外）
- **网络出站访问控制**：SkillApp 的网络出站请求应受到权限声明约束，用户可查看和控制每个 SkillApp 的网络访问范围，防止 Skill 恶意外传本地数据

### 可用性

- **操作引导**：面向非技术用户，所有关键操作应有清晰的引导提示，无需阅读文档即可上手
- **错误提示友好**：生成失败、编译错误等情况应给出用户可理解的错误说明，而非技术堆栈信息
- **可访问性**：界面应满足基本的无障碍访问标准（键盘导航、对比度等）

### 跨平台兼容性

- **宿主 OS 支持**：优先支持 macOS 和 Windows；Linux 作为次优先支持平台
- **生成应用兼容性**：由 IntentOS 生成的 SkillApp 应在宿主 OS 上原生运行，无需额外依赖环境
- **架构支持**：支持 x86_64 和 Apple Silicon（ARM64）架构

### 可扩展性

- **Skill 数量支撑**：系统应支持管理 50 个以上已安装 Skill，Skill 数量增长不应导致 Skill 管理中心的响应时间线性退化
- **SkillApp 累计数量**：系统应支持用户累计生成并管理 20 个以上 SkillApp，不因数量增长出现显著性能下降
- **动态扩展能力**：架构设计应允许后续以增加 Skill 的方式扩展系统能力，无需修改 IntentOS 核心层（对应「动态生长」设计理念）

### 可维护性

- **日志与诊断**：系统运行日志、SkillApp 生成日志可供用户或技术支持人员查阅
- **更新机制**：IntentOS Desktop 本身支持自动检测和安装更新

## 5. 约束条件

### 技术依赖约束

| 约束项 | 说明 |
|--------|------|
| **OpenClaw 版本** | IntentOS 内核层强依赖 OpenClaw 框架；系统设计需与 OpenClaw 提供的规划引擎和代码生成引擎接口兼容，版本升级需经过回归验证 |
| **Electron 版本** | IntentOS Desktop 和生成的 SkillApp 均基于 Electron 构建；需指定最低 Electron 版本（建议跟随 Electron 稳定版本），且 Desktop 与 SkillApp 使用的 Electron 版本可以不同 |
| **Node.js 版本** | Electron 内置 Node.js 运行时，代码生成和编译工具链依赖 Node.js 版本需与 Electron 版本配套 |
| **MCP 协议版本** | 宿主 OS 资源访问通过 MCP 协议实现，系统需声明兼容的 MCP 协议版本范围，版本变更需保持向后兼容 |

### 宿主 OS 兼容范围

| 平台 | 最低版本 | 说明 |
|------|----------|------|
| macOS | macOS 12 Monterey | 支持 Intel 和 Apple Silicon 双架构 |
| Windows | Windows 10 21H2 | 64 位系统 |
| Linux | Ubuntu 20.04 LTS | 仅次优先，社区支持为主 |

### 架构约束

- **进程隔离**：每个 SkillApp 必须运行在独立 Electron 进程中，不允许共享渲染进程
- **资源访问路径唯一性**：所有底层资源访问必须经过 MCP 接口，不允许 SkillApp 直接绕过 IntentOS 层调用系统 API
- **Skill 原子性**：Skill 是系统最小可复用单位，单个 Skill 内部逻辑不可再拆分；Skill 间通信必须通过 OpenClaw 提供的调度接口进行

### 设计约束

- **无代码生成界面**：IntentOS Desktop 本身不包含代码编辑器，用户不应在管理台界面直接看到或修改生成的代码（开发者工具除外）
- **意图驱动优先**：所有 SkillApp 的生成和修改必须以用户意图为起点，不支持直接导入外部代码替代生成流程
- **单一管理入口**：所有 SkillApp 的生命周期管理（启动、修改、卸载）统一由 IntentOS Desktop 管理中心操作，不允许用户绕过管理台直接操作 SkillApp 进程
